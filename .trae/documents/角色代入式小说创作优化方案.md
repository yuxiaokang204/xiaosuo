# 角色代入式小说创作优化方案

## 概述

重构小说创作流程，实现"先设计世界观/故事走向/角色性格/外观/背景，再让大模型多线程并行代入每个角色进行创作"的核心能力。

**核心机制**：
1. **角色经历记忆链**（保证人物行为一致）：每个角色拥有自己的经历记忆，记录其经历的所有重要事件、情绪变化、认知更新、性格演变。角色每次代入时携带完整经历记忆，行为基于"原始性格 + 经历塑造的认知"综合决策，不会"失忆"或做出与经历矛盾的行为。
2. **多线程并行角色代入**：所有出场角色同时代入（asyncio.gather），无需串行等待。
3. **角色生命周期管理**：角色有出场和退场，退场后不再出现。
4. **三种代入策略**：角色驱动写作（默认）/ 角色视角章节 / 角色扮演对话生成，自动切换。

**保证小说不混乱的关键**：角色经历记忆链确保每个角色的行为、语言、决策、情绪都与其设定和过往经历一致，避免角色"失忆"、行为突变、信息泄露等问题。

## 当前状态分析

### 现有架构
- **6-Skill协作**：故事架构师、世界观构建师、角色塑造师、开篇钩子师、专业写手、文风精修师
- **三阶段循环**：SKELETON(骨架) → DETAIL(细节) → POLISH(精修)
- **数据流**：worldbuilding → characters → outlining → drafting

### 核心问题（与用户诉求的差距）

| 问题 | 现状 | 影响 |
|------|------|------|
| 数据结构不完整 | CharacterDB缺失 psychological_profile/behavior_tags/relationship_webs/speech_fingerprint；WorldSettingDB缺失 key_locations/factions/unique_appeal | Agent生成的丰富细节持久化时丢失，后续章节无法复用 |
| 角色信息截断 | chapter_pipeline.py:847 `characters = context.get("characters", "")[:400]` | 完整角色档案2000+字被截断到400字符，语言指纹/行为标签几乎必然丢失 |
| 无代入机制 | LLM始终以"作者"身份写作，角色是被描述的对象 | 角色言行可能不符合设定，缺乏沉浸感 |
| 角色世界观割裂 | Loop 0并行生成时角色agent收到 world_info=""；outlining只传角色名字列表 | 角色与世界观脱钩，故事走向无法基于角色性格设计 |
| 无角色生命周期 | 所有角色全程存在，无出场/退场概念 | 无法实现"角色不再出现即消失" |
| 无角色记忆链 | state_tracker仅跟踪当前位置/情绪/物品，无"该角色经历过的互动摘要" | 角色每次出现都是"失忆"状态，无法携带之前的记忆 |

#### 3.2 多线程并行角色代入（[src/backend/core/chapter_pipeline.py](file:///d:/trae/novel-agent-system-v1.1.0/src/backend/core/chapter_pipeline.py)）

在 `_agent_character` 之后、`_agent_draft` 之前新增 `_agent_character_roleplay_parallel` 方法：

```python
async def _agent_character_roleplay_parallel(
    self, chapter_idx: int, title: str, context: Dict, loop_metadata: Dict
) -> AgentContribution:
    """并行代入所有出场角色"""
    # 1. 从大纲中提取本章出场角色（基于角色生命周期过滤）
    active_characters = self._get_active_characters_for_chapter(chapter_idx)
    
    # 2. 识别场景类型
    scene_type = self._detect_scene_type(context.get("outline", ""))
    
    # 3. 并行为每个角色生成代入卡
    roleplay_tasks = [
        self._roleplay_single_character(char, context, chapter_idx, scene_type)
        for char in active_characters
    ]
    roleplay_cards = await asyncio.gather(*roleplay_tasks, return_exceptions=True)
    
    # 4. 汇总代入卡
    valid_cards = [c for c in roleplay_cards if not isinstance(c, Exception)]
    return AgentContribution(
        agent_name="角色代入师",
        content=self._format_roleplay_cards(valid_cards),
        score=8.0,
        details={"character_count": len(valid_cards), "scene_type": scene_type}
    )
```

#### 3.3 角色生命周期过滤

```python
def _get_active_characters_for_chapter(self, chapter_idx: int) -> List[Dict]:
    """获取本章活跃角色（已出场且未退场）"""
    active = []
    for char in self.state.characters:
        first = char.get("first_appear_chapter", 1)
        last = char.get("last_appear_chapter", None)
        if first <= chapter_idx and (last is None or chapter_idx <= last):
            active.append(char)
    return active
```

#### 3.4 角色经历记忆链管理（[src/backend/core/state_tracker.py](file:///d:/trae/novel-agent-system-v1.1.0/src/backend/core/state_tracker.py)）

扩展 StateTracker，为每个角色维护**经历记忆链**（保证人物行为一致性的核心机制）：

```python
@dataclass
class CharacterExperienceMemory:
    """角色经历记忆 — 保证人物行为一致性"""
    chapter: int
    experienced_events: str        # 该角色在此章经历的所有重要事件（第一人称视角）
    emotional_trajectory: str      # 情绪变化轨迹（如"平静→震惊→愤怒→释然"）
    cognition_updates: List[str]   # 认知更新：新获得的认知/信念改变
    personality_shifts: str        # 性格微调：因此章经历产生的性格变化
    decisions_made: List[str]      # 该角色在此章做出的关键决策
    information_gained: List[str]  # 该角色在此章获得的关键信息
    relationships_change: Dict[str, str]  # 该角色对他人看法的改变

class StateTracker:
    def __init__(self):
        # ... 现有字段 ...
        self._character_memories: Dict[str, List[CharacterExperienceMemory]] = {}
    
    def get_character_memory_chain(self, name: str, up_to_chapter: int) -> List[CharacterExperienceMemory]:
        """获取角色截至某章的完整经历记忆链
        
        用于角色代入时注入，保证角色行为基于其过往经历形成的认知，
        而非"失忆"状态。这是保证人物行为一致性的核心。
        """
        chain = self._character_memories.get(name, [])
        return [m for m in chain if m.chapter < up_to_chapter]
    
    def append_character_memory(self, name: str, entry: CharacterExperienceMemory):
        """章节生成后追加角色经历记忆"""
        self._character_memories.setdefault(name, []).append(entry)
    
    def get_character_behavior_context(self, name: str, up_to_chapter: int) -> str:
        """生成角色行为上下文摘要（用于代入卡生成）
        
        将角色经历记忆链整理为行为指导：
        - 过往经历如何塑造了角色当前认知
        - 角色当前的情绪状态和性格倾向
        - 角色已知和未知的信息
        - 角色对其他人物的当前看法
        """
        chain = self.get_character_memory_chain(name, up_to_chapter)
        if not chain:
            return "（该角色尚无过往经历）"
        # 整理为行为一致性指导文本
        ...
```

**记忆链如何保证人物行为一致**：
1. **认知连续性**：角色不会"忘记"自己经历过的事，后续决策基于完整经历
2. **性格演变**：角色性格会随经历微调（如经历背叛后变得多疑），但演变有迹可循
3. **信息边界**：角色只会基于自己"应该知道"的信息行动，不会泄露剧情信息
4. **情绪延续**：角色进入新场景时的情绪状态承接上一章结尾

#### 3.5 三种代入策略实现

**策略选择逻辑**（`_detect_scene_type`）：
- **perspective（角色视角章节）**：大纲标记为"角色视角"或前3章后的特定转折章节 → 从主角第一人称视角写作
- **dialogue（对话密集场景）**：大纲中包含"对话/谈判/冲突/质问"等关键词 → 触发角色扮演对话生成
- **narrative（叙事主导，默认）**：标准第三人称叙事 + 角色代入卡注入

**角色视角章节实现**：
- DraftAgent 的 system prompt 切换为第一人称视角
- 注入主角的完整代入卡作为"你的身份"
- 输出仍为小说正文，但视角为主角第一人称

**角色扮演对话生成实现**：
- 识别对话密集场景后，先让每个角色"扮演"自己生成对话
- 再由 DraftAgent 整合为叙事文本
- 对话生成时注入所有参与角色的代入卡

```
5. decisions_made: 你做出了哪些关键决策？
6. information_gained: 你获得了哪些关键信息？（影响后续行为）
7. relationships_change: 你对其他人物的看法有何改变？

严格基于本章正文内容生成，不要编造未发生的事件。
```

生成的经历记忆条目：
1. 追加到 `StateTracker._character_memories`（运行时使用）
2. 持久化到 `CharacterMemoryDB`（跨会话使用）
3. 用于该角色下次代入时的行为一致性保障

### 阶段五：一致性保障（核心：基于经历记忆的行为一致性）

#### 5.1 角色行为一致性校验（[src/backend/core/consistency_checker.py](file:///d:/trae/novel-agent-system-v1.1.0/src/backend/core/consistency_checker.py)）

**核心目标：保证人物行为一致**。扩展校验项：
- **性格一致性**：角色行为是否符合 personality 设定
- **语言一致性**：角色对话是否符合 speech_fingerprint（口头禅/语速/禁忌词）
- **行为标签一致性**：角色行为是否体现 behavior_tags
- **决策模式一致性**：角色决策是否符合 psychological_profile.decision_pattern
- **经历记忆一致性**（核心新增）：
  - 角色是否"失忆"——提及了该角色不应该知道的信息
  - 角色行为是否与其经历记忆塑造的认知矛盾
  - 角色情绪状态是否承接上一章结尾
  - 角色性格变化是否有经历支撑（不能突变）
- **信息边界一致性**：角色只会基于自己"应该知道"的信息行动

#### 5.2 角色生命周期校验

- 校验已退场角色不再出现在后续章节
- 校验角色出场章节与大纲一致
- 校验角色经历记忆链无断层（每章都有记忆条目）

#### 5.3 单主角规则保持

- 角色视角章节仅限主角第一人称
- 故事走向以主角成长弧线为核心
- 配角代入卡服务于主角叙事，不喧宾夺主
- 配角的经历记忆围绕与主角的交集展开

## 假设与决策

### 假设
1. 现有 SQLite 数据库可通过新增字段平滑迁移（使用 default=None 保证向后兼容）
2. LLM 能力足以支撑"角色代入"（需要较强的角色扮演能力，建议使用 temperature=0.8）
3. 并行角色代入的 LLM 调用不会触发速率限制（单章通常2-5个出场角色）

### 决策
1. **角色代入作为独立Agent**：而非修改现有 CharacterAgent，保持职责单一
2. **记忆链存储在 StateTracker + CharacterMemoryDB**：StateTracker 用于运行时快速访问，CharacterMemoryDB 用于持久化
3. **三种代入策略自动选择**：基于大纲关键词识别场景类型，无需用户手动指定
4. **Loop 0 改为串行**：牺牲少量速度换取角色↔世界观↔故事走向的强关联
5. **保留单主角规则**：角色视角章节仅限主角，配角代入卡服务于主角叙事
